# JSK 일정 관리 서비스 - 이슈 처리 팀

자동으로 발생하는 문제(배포 에러, 코드 품질, 보안 등)를 감지하고 팀 단위로 병렬 처리하는 에이전트 팀입니다.

---

## 팀 구성 및 역할

### 1️⃣ 이슈 분석 단계 (Designer)

**목표**: 발생한 이슈를 분류하고 영향 범위 파악

**담당 작업**:
- 에러 메시지 분석 (배포, 테스트, 린트 등)
- 근본 원인 파악 및 영향 범위 확인
- 해결 전략 제시 (필요한 수정 사항)
- 작업 분배 (Developer와 QA에 할당할 작업 명시)

**출력 형식**:
```
## 이슈 분석 결과
- **분류**: [배포 에러 / 코드 버그 / 성능 / 보안 / 기타]
- **심각도**: [Critical / High / Medium / Low]
- **원인**: [구체적 원인]
- **영향 범위**: [영향받는 파일/기능]
- **해결 방안**:
  - Developer 작업: [구체적 수정 사항]
  - QA 작업: [테스트 시나리오]
```

---

### 2️⃣ 병렬 처리 단계

#### 🛠️ Developer - 코드 수정
**목표**: 분석 결과에 따라 코드를 수정하고 배포 준비

**담당 작업**:
- 이슈 원인에 해당하는 코드 수정
- 관련 테스트 코드 추가/수정 (필요시)
- 로컬 테스트 완료 후 결과 보고
- 커밋 메시지 작성 (명확한 설명)

**체크리스트**:
- [ ] 코드 수정 완료
- [ ] 로컬 빌드/테스트 통과
- [ ] 보안 검토 (hardcoding, XSS 등)
- [ ] 커밋 메시지 작성

---

#### 🧪 QA - 코드 품질 검토 및 테스트
**목표**: 수정된 코드의 품질을 검증하고 회귀 테스트 실행

**담당 작업**:
- 수정된 코드 리뷰 (스타일, 컨벤션, 보안)
- 제안된 테스트 시나리오 실행
- 기존 기능 회귀 테스트
- 발견된 이슈가 있으면 Developer에게 피드백

**체크리스트**:
- [ ] 코드 품질 검토 완료
- [ ] 테스트 시나리오 실행 (정상/예외/경계값)
- [ ] 회귀 테스트 통과
- [ ] 최종 승인 또는 수정 요청

---

### 3️⃣ 최종 검증 및 배포

**담당**: Developer (QA 승인 후)

**작업**:
- 커밋 & 푸시 (GitHub)
- 배포 상태 모니터링
- 배포 완료 후 결과 보고

---

## 자동 호출 트리거

이 팀은 다음 키워드 감지 시 자동으로 활성화됩니다:

- **배포/병합 에러**: "deployment failed", "build error", "push failed"
- **코드 버그**: "bug", "error", "exception", "fail"
- **성능/보안**: "performance", "vulnerability", "security", "timeout"
- **테스트 실패**: "test failed", "assertion error"
- **린트 에러**: "lint error", "style error", "format"

---

## 실행 흐름

```
입력 (이슈 발생)
    ↓
[Designer] 이슈 분석 & 전략 수립
    ↓
  ┌─────────────────────┬──────────────────────┐
  ↓                     ↓                      ↓
[Developer]         [QA]              (독립 병렬 처리)
코드 수정 & 테스트  품질 검토 & 테스트
  ↓                     ↓
  └─────────────────────┴──────────────────────┘
    ↓
[검증] 모든 작업 완료 확인
    ↓
[배포] Git commit & push
    ↓
완료 (자동 배포)
```

---

## 팀 협업 규칙

### 1. 의존성 관리
- **Developer → QA**: Developer 완료 후 QA 시작
- **QA → Developer**: 이슈 발견 시 Developer에게 피드백 (1회 순환)
- **최종 배포**: QA 승인 후 Developer가 푸시

### 2. 커뮤니케이션
- 각 단계 완료 후 **명확한 결과 보고**
- 이슈 발견 시 **재현 방법과 함께 보고**
- 불명확한 사항은 **사용자에게 확인**

### 3. 품질 표준
- **코드 스타일**: CLAUDE.md 컨벤션 준수
- **테스트 커버리지**: 정상/예외/경계값 모두 포함
- **보안**: OWASP Top 10 취약점 자체 검토
- **성능**: 필요시 병목 분석 및 최적화

## 작업 예시

### 예시 1: 배포 에러
```
🔴 에러: "Build failed: Module not found"

[Designer] → "frontend에서 import 경로 오류 분석"
            → "Developer: 경로 수정, QA: 관련 파일 테스트"

[Developer] → "import 경로 수정 & 로컬 빌드 테스트 ✅"

[QA] → "번들 확인, 런타임 테스트 ✅"

✅ 완료: git commit & push
```

### 예시 2: 코드 버그
```
🔴 이슈: "일정 삭제 시 알림이 발송되지 않음"

[Designer] → "Edge Function 호출 실패 분석"
            → "Developer: 호출 로직 수정, QA: 통합 테스트"

[Developer] → "Edge Function 호출 추가 & 테스트 ✅"

[QA] → "일정 삭제 → 알림 발송 확인 ✅"

✅ 완료: git commit & push
```

## 팀의 자동 활성화 조건

1. **입력**: 에러 메시지, 버그 리포트, 성능 이슈 등
2. **조건**: 키워드 감지 (위 "자동 호출 트리거" 참고)
3. **실행**: Designer → (Developer || QA) → 배포
4. **결과**: 이슈 해결 + 자동 배포

이 팀은 **자동으로 활성화되므로** 별도의 호출이 필요하지 않습니다.
