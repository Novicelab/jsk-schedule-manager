## 1. 서비스 개요

### 1.1 서비스 목적

JSK 일정 관리 서비스는 팀/그룹 단위의 일정을 공유하고 협업할 수 있는 웹 기반 일정 관리 서비스입니다. 개인 일정 관리 도구가 아닌 **팀 중심의 일정 관리**를 핵심 가치로 하며, 팀원 간 일정 가시성 확보와 효율적인 협업을 목표로 합니다.

### 1.2 타겟 사용자

| 구분 | 설명                                        |
|------|-------------------------------------------|
| 주 타겟 | 소규모~중규모 팀(5~30명)에서 일정 공유가 필요한 조직          |
| 사용 환경 | 웹 브라우저 기반 (모바일 우선)                        |
| 사용 시나리오 | 팀 일정 등록/공유, 팀원 휴가 현황 파악, 일정 변경 시 즉시 알림 수신 |

### 1.3 핵심 가치 제안

1. **팀 일정 가시성**: 팀원 전체가 누가 언제 무엇을 하는지 한눈에 파악
2. **실시간 알림**: 일정 변경(등록/수정/삭제) 시 카카오톡으로 즉시 알림
3. **휴가 관리 통합**: 개인 휴가와 팀 일정을 하나의 캘린더에서 관리
4. **팀 간 일정 조회**: 팀에 소속되지 않은 사용자도 타 팀 일정 열람 가능

## 4. 기능 명세

### 4.1 회원가입 (F-AUTH-01)

**기능명**: 회원가입
**목적**: 카카오톡 OAuth를 통한 간편 계정 생성

**상세 요구사항**:
- 카카오톡 OAuth 2.0 기반 소셜 로그인으로 가입 처리
- 카카오 계정에서 제공하는 정보(이름, 카카오톡 ID) 자동 수집
- 최초 가입 시 서비스 이용 약관 동의 절차 포함
- 가입 완료 후 자동으로 로그인 상태로 전환
- 카카오톡 연동 정보는 알림 발송에 그대로 활용

**예외/제약사항**:
- 카카오 OAuth 인증 실패 시 오류 메시지 반환
- 카카오 서버 장애 시 서비스 로그인 불가 — 운영 정책 수립 필요
- 동일 카카오 계정으로 중복 가입 불가 (기존 계정으로 자동 로그인 처리)

---

### 4.2 로그인 / 로그아웃 (F-AUTH-02)

**기능명**: 로그인 / 로그아웃
**목적**: 인증된 사용자만 서비스 이용 가능하도록 접근 제어

**상세 요구사항**:
- 카카오톡 OAuth 2.0을 통한 로그인
- 로그인 성공 시 JWT 토큰(Access Token + Refresh Token) 발급
- 로그인 상태는 마지막 로그인 기준 30일간 유지 (Refresh Token 기반)
- 로그아웃 시 토큰 무효화
- 인증되지 않은 사용자의 API 접근 차단

**예외/제약사항**:
- 카카오 OAuth 인증 실패 시 로그인 불가 처리
- Access Token 만료 시 Refresh Token으로 자동 재발급
- Refresh Token 만료(30일 경과) 시 재로그인 유도
- 로그아웃 후 토큰 재사용 시 401 오류 반환

---

### 4.3 팀 생성 (F-TEAM-01)

**기능명**: 팀 생성
**목적**: 일정을 공유할 팀 단위 생성

**상세 요구사항**:
- 필수 입력: 팀 이름
- 선택 입력: 팀 설명
- 팀 생성자는 자동으로 해당 팀의 Admin 역할 부여
- 한 사용자가 여러 팀에 소속 가능
- 한 사용자가 여러 팀을 생성 가능

**예외/제약사항**:
- 팀 이름은 1~50자 제한
- 팀 이름 중복 허용 여부 — 설계 시 결정 필요
- 비로그인 사용자 및 팀 미소속 회원(User)은 팀 생성 불가
- 팀 생성 권한: Member 또는 Admin만 가능 (최소 1개 팀에 소속되어 있어야 함)

---

### 4.4 팀원 초대 / 추방 (F-TEAM-02)

**기능명**: 팀원 초대 및 추방
**목적**: 팀 구성원 관리

**상세 요구사항**:
- Admin만 팀원 초대/추방 가능
- 초대: 카카오톡으로 팀 초대 링크 발송 (카카오톡 메시지 API 활용)
- 초대 링크 클릭 시 서비스 페이지로 이동하여 수락/거절 선택 가능
- 추방: Admin이 특정 멤버를 팀에서 제거
- 추방된 사용자에게 카카오톡 알림 발송

**예외/제약사항**:
- 이미 팀에 소속된 사용자에게 중복 초대 불가
- 초대 링크는 유효 기간 설정 필요 — 설계 시 결정 필요 (예: 24시간)
- Admin 자기 자신을 추방할 수 없음
- 팀에 Admin이 1명뿐인 경우, 해당 Admin은 추방/탈퇴 불가 (팀에 Admin 최소 1명 유지)
- 미가입(카카오 미연동) 사용자에 대한 초대 처리 방식 — 설계 시 결정 필요

---

### 4.5 일정 등록 (F-SCH-01)

**기능명**: 일정 등록
**목적**: 팀 일정 또는 개인 휴가를 등록하여 팀원과 공유

**상세 요구사항**:
- 필수 입력: 제목, 일정 유형(팀 일정/개인 휴가), 시작 날짜/시간, 종료 날짜/시간
- 선택 입력: 설명
- 일정 유형: 개인 휴가 / 팀 일정
- 등록 즉시 해당 팀 전체 구성원에게 카카오톡 알림 발송
- 종일 일정 지원 (시간 미지정 시 종일로 처리)

**예외/제약사항**:
- 종료 날짜/시간이 시작 날짜/시간보다 앞설 수 없음
- 팀에 소속되지 않은 사용자는 일정 등록 불가
- 제목은 1~100자 제한
- 과거 날짜에 일정 등록 허용 여부 — 설계 시 결정 필요

---

### 4.6 일정 조회 (F-SCH-02)

**기능명**: 일정 조회
**목적**: 팀 일정 현황 파악

**상세 요구사항**:
- 팀원: 소속 팀의 전체 일정 조회 가능
- 팀 미소속 사용자: 타 팀의 일정 조회 가능
- 목록 조회: 날짜 범위 기준 필터링
- 상세 조회: 개별 일정의 제목, 일시, 설명, 등록자, 일정 유형 확인
- 아카이브(삭제)된 일정은 일반 조회에서 제외

**예외/제약사항**:
- 비로그인 사용자는 조회 불가
- 타 팀 일정 조회 시 일정 수정/삭제 버튼 비노출
- 아카이브된 일정 조회는 Admin 전용 기능으로 분리

---

### 4.7 일정 수정 (F-SCH-03)

**기능명**: 일정 수정
**목적**: 변경된 일정 정보 반영

**상세 요구사항**:
- Member: 본인이 등록한 일정만 수정 가능
- Admin: 팀 내 모든 일정 수정 가능
- 수정 가능 필드: 제목, 일정 유형, 시작/종료 날짜/시간, 설명
- 수정 즉시 해당 팀 전체 구성원에게 카카오톡 알림 발송
- 수정 이력은 별도 관리하지 않음 (V1 기준)

**예외/제약사항**:
- 권한 없는 사용자의 수정 시도 시 403 오류 반환
- 아카이브된 일정은 수정 불가
- 수정 시에도 종료 시간이 시작 시간보다 앞설 수 없음

---

### 4.8 일정 삭제 (F-SCH-04)

**기능명**: 일정 삭제 (소프트 딜리트)
**목적**: 불필요한 일정 정리 (복구 가능한 형태)

**상세 요구사항**:
- Member: 본인이 등록한 일정만 삭제 가능
- Admin: 팀 내 모든 일정 삭제 가능
- 삭제 시 물리 삭제가 아닌 소프트 딜리트 (`deleted_at` 타임스탬프 기록)
- 삭제 즉시 해당 팀 전체 구성원에게 카카오톡 알림 발송

**예외/제약사항**:
- 권한 없는 사용자의 삭제 시도 시 403 오류 반환
- 이미 아카이브된 일정은 재삭제 불가
- 아카이브된 일정 복구 기능 — V1에서는 미지원

---

### 4.9 카카오톡 알림 발송 (F-NTF-01)

**기능명**: 카카오톡 알림 발송
**목적**: 일정 변경 사항 및 팀 초대를 팀원에게 즉시 전달

**상세 요구사항**:
- 알림 발송 트리거 및 대상:
  - 일정 등록/수정/삭제 시 → 해당 팀 전체 구성원에게 발송
  - 팀 초대 시 → 초대 대상자에게 초대 링크 포함하여 발송
  - 팀 추방 시 → 추방된 사용자에게 발송
- 알림 내용 (일정 CRUD): 일정 제목, 변경 유형(등록/수정/삭제), 변경자 이름, 일정 날짜
- 알림 내용 (팀 초대): 팀 이름, 초대자 이름, 초대 수락 링크
- 카카오톡 알림톡 API 활용 (OAuth로 연동된 카카오 계정 기반)

**예외/제약사항**:
- 카카오톡 API 호출 실패 시 재시도 로직 필요 (최대 3회)
- 알림 발송 실패가 일정 CRUD 트랜잭션에 영향을 주지 않아야 함 (비동기 처리)
- 카카오 OAuth 가입 사용자이므로 카카오톡 연동은 기본 보장 (별도 연동 처리 불필요)
- 알림 발송 이력 저장 여부 — 설계 시 결정 필요

---

### 4.10 캘린더 뷰 (F-CAL-01)

**기능명**: 캘린더 뷰
**목적**: 일정을 시각적으로 파악

**상세 요구사항**:
- 월간 뷰: 해당 월의 전체 일정을 달력 형태로 표시
- 주간 뷰: 해당 주의 일정을 시간대별로 표시
- 일간 뷰: 해당 일의 일정을 시간대별로 상세 표시
- 일정 유형별 색상 구분 (개인 휴가 / 팀 일정)
- 캘린더에서 날짜 클릭 시 해당 날짜의 일정 목록 표시
- 뷰 간 전환 가능 (월간 ↔ 주간 ↔ 일간)

**예외/제약사항**:
- 일정이 없는 날짜는 빈 상태로 표시
- 하루에 표시할 수 있는 일정 개수 초과 시 "+N개 더보기" 처리
- 아카이브된 일정은 캘린더에 표시하지 않음

## 6. 다음 단계(설계)를 위한 핵심 결정 사항

### 6.1 기술 관련 결정

| 결정 항목 | 선택지 | 권장 | 사유 |
|-----------|--------|------|------|
| 인증 방식 | 세션 기반 vs JWT 토큰 | JWT | REST API 기반 서비스에 적합, 추후 모바일 확장 대비 |
| 프론트엔드 | React (FullCalendar) | **확정** | 캘린더 SPA UX, FullCalendar 통합 |
| 데이터베이스 | Supabase (PostgreSQL) | **확정** | BaaS, 클라우드 관리형 DB |
| 빌드 도구 | Gradle | **확정** | Spring Boot 기본 권장, 빌드 속도 우수 |
| 캘린더 라이브러리 | FullCalendar | **확정** | React 통합, 캘린더 UI 라이브러리 |
| 배포 플랫폼 | Render | **확정** | 클라우드 앱 호스팅, GitHub 연동 자동 배포 |
| 소스관리/CI·CD | GitHub | **확정** | 코드 버전관리 및 Render 자동 배포 트리거 |

### 6.2 정책 관련 미결정 사항

| 결정 항목 | 설명 |
|-----------|------|
| 팀 이름 중복 허용 | 동일한 팀 이름을 허용할지 여부 |
| 과거 날짜 일정 등록 | 지나간 날짜에 일정 등록 가능 여부 |
| 카카오 미연동 사용자 초대 | 카카오톡 계정이 없는 사용자 초대 시 처리 방식 |
| 초대 링크 유효 기간 | 팀 초대 링크의 만료 시간 설정 (예: 24시간, 7일) |
| 알림 발송 이력 저장 | 알림 발송 기록을 DB에 저장할지 여부 |
| 타 팀 일정 조회 범위 | 타 팀 일정의 상세 정보 공개 범위 (제목만 vs 전체) |
| 한 사용자의 최대 팀 소속 수 | 제한 없음 vs 상한선 설정 |

### 6.3 설계 단계 진입 시 필요한 산출물

1. **ERD**: User, Team, TeamMember, Schedule, Notification 등 엔티티 설계
2. **API 설계서**: RESTful API 엔드포인트 목록 및 요청/응답 스펙
3. **시스템 아키텍처**: 전체 시스템 구조도 (프론트엔드-백엔드-DB-외부API)
4. **화면 설계**: 주요 화면 와이어프레임 (캘린더 뷰, 일정 등록 폼, 팀 관리 화면 등)

### 6.4 기능 의존성 맵

```
회원가입(F-AUTH-01)
  └── 로그인(F-AUTH-02)
        ├── 팀 생성(F-TEAM-01)
        │     └── 팀원 초대/추방(F-TEAM-02)
        │           └── 일정 등록(F-SCH-01)
        │                 ├── 일정 조회(F-SCH-02)
        │                 ├── 일정 수정(F-SCH-03)
        │                 ├── 일정 삭제(F-SCH-04)
        │                 └── 카카오톡 알림(F-NTF-01) ← 등록/수정/삭제 시 트리거
        │
        └── 캘린더 뷰(F-CAL-01) ← 일정 데이터 의존
              └── 주간/일간 뷰(F-CAL-02)
```
